# Material Frontier Online — クライアントとサーバーの責務

- 文書状態: レビュー草案
- 最小試作: 完全オフライン。ネットワークサーバーは作らない
- 設計目的: 将来のオンライン化を妨げない責務境界だけを先に置く
- 今回の改訂根拠: `origin: user_explicit`
- Gate 0: 閉鎖中。本文書の改訂はネットワークまたはゲーム本体の実装承認ではない

## 1. 最小試作の構成

> origin: `user_explicit`（オフライン試作、ネットワーク非実装）、`user_approved`（ローカル権威境界）

```text
Input Device
  ↓
Input Adapter
  ↓ InputCommand
Local Authority Simulation
  ↓ DomainEvent / Read-only State
Presentation
  ├─ Animation / VFX / Lighting
  ├─ SFX / BGM
  ├─ HUD / Result
  └─ Camera
```

すべて同一プロセスで動作する。`Local Authority Simulation` は将来のサーバー相当の責務を持つが、ソケット、パケット、認証、ヘッドレスサーバーは実装しない。

## 2. プレイヤーデータ5層の責務

> origin: `user_explicit`

| 層 | 最小試作での責務 | 戦闘への関与 | 疲労破壊時 |
|---|---|---|---|
| `AvatarIdentity` | 名前、顔、髪、声、人格等を表す | なし | 変更・削除しない |
| `MaterialJob` | 材料系統、基本物性、魔法適性、耐性、壊れ方を選ぶ | `MaterialSnapshot` の入力 | 定義・既知設計を削除しない |
| `CombatForm` | 操作、移動補正、アクションセット、判定形状を選ぶ | 入力コマンドを実行可能アクションへ解決 | 定義を削除しない |
| `EquipmentInstance` | 一つの装備個体を表す。将来は加工履歴、疲労履歴、個体ツリーも保持できる | MVPではcore個体の損傷状態だけをローカル権威シミュレーションが更新 | 将来、当該個体とその加工・個体スキルツリーを使用不能にする |
| `Appearance` | 鎧・武器の外装、主色、副色、発光色等を表す | 表示のみ | 外装定義自体は削除しない |

素材が性能と壊れ方を決め、`CombatForm` が操作とアクションセットを決める。入力層、表示層、装備個体のいずれも、この境界を越えて素材専用の戦闘コードを持たない。

最小試作では5層を永続サービスへ保存せず、セッション開始時に試験用データから構築する。5層へ分けることは、アカウントやデータベースを先行実装する理由にはしない。

### 2.1 `EquipmentLoadout`と損傷所有権

`EquipmentLoadout` は装備個体を束ねる集約ルートであり、`core_equipment_instance_id` と `equipment_instance_ids[]` を持つ。最小試作では装備個体を正確に1個だけ持ち、それをcore装備とする。

- core装備だけがプレイヤーの `Integrity`、`Deformation`、`temperature` を所有する。
- `ActorRuntimeState.core_equipment_runtime_ref` が、そのcore装備の実行状態を指す。
- Actor自身、独立武器、独立鎧へ同じ損傷値を重複保持しない。
- 武器耐久、鎧耐久、複数装備スロット、被弾箇所に基づく損傷分配は将来範囲である。

最小試作の現在構成については、core `EquipmentInstance.material_job_id` と `combat_form_id` を唯一の書き換え可能な正とする。Actor側にはcore装備から解決した読み取り専用参照だけを置く。ローカル権威シミュレーション開始前に、core参照、所有者、MaterialJob、CombatForm、許可された組み合わせ、およびActorの解決済み参照との一致を検証する。不一致を暗黙補正せず、構成エラーとして開始を拒否する。

## 3. オフライン試作の責務

> origin: `user_explicit`（損傷・Heat/BurnCurse・実装範囲）、`user_approved`（表示／シミュレーション分離）

### 3.1 入力・表示層

- キーボード、マウス、ゲームパッド等の読み取り
- 入力を `InputCommand` へ変換
- スプライト、アニメーション
- VFX、端末側パーティクル、破片、残光
- リアルタイム光源の表示
- BGM、SE
- HUD、操作表示、リザルト
- カメラ、補間、画面振動
- 画質設定による装飾削減
- 開発用性能表示

表示層は、HP、部位HP、報酬、ギミック結果を直接決定しない。

`Appearance` はここで読み取るが、配色、外装、発光色からダメージ、耐性、判定範囲を逆算しない。素材固有の損傷音や熱表示は、権威状態から発行されたイベントを表現へ変換する。

### 3.2 ローカル権威シミュレーション

- プレイヤーと敵の正規位置・速度
- 移動とゲームプレイ衝突
- 行動可能性
- 予備動作、実判定、後隙
- クールダウンと資源消費
- 命中、ダメージ、装備個体の `Integrity`、`Deformation`、状態異常
- `Heat`、`BurnCurse`、腐食の独立したチャンネル処理
- 温度・腐食の低頻度更新
- `MaterialJob` から戦闘性能と損傷挙動を、`CombatForm` から操作・アクションセットを解決
- core `EquipmentInstance` の現在加工状態の読み取り
- 敵AIとフェーズ
- 部位HPと破壊状態
- ギミック状態
- HP0判定
- 死体生成と剥ぎ取り権利
- 報酬乱数と報酬付与
- リトライ時の状態初期化

`Heat` は温度上昇、軟化、溶融、熱衝撃等を扱い、`BurnCurse` は育成、加工、組織、耐性を崩す呪いを扱う。炎耐性、耐熱性、バーン耐性は別々に参照し、片方の結果をもう片方へ自動転用しない。

疲労破壊を有効化する将来段階では、権威シミュレーションが対象 `EquipmentInstance` を破壊済みにし、その個体の加工状態と個体スキルツリーを使用不能にする。`AvatarIdentity` を削除、初期化、作り直しの対象にしない。

`processing_history`、`fatigue_history`、`instance_skill_tree_state`、`EquipmentFatigueChanged`、`EquipmentDestroyed` は将来予約である。最小試作では生成、更新、保存、表示、イベント発行しない。core装備の `Integrity == 0` は `ActorDefeated` として処理し、装備破壊イベントを発行しない。

### 3.3 定義データ・検証層

- 定義データの読み込み
- IDと参照の検証
- 素材ステータスの事前計算
- `CombatFormDefinition` とアクションセットの参照検証
- `EquipmentLoadout` がcore装備を正確に1個持ち、その所有者と参照が有効であることの検証
- core装備の `MaterialJob`／`CombatForm` とActorの読み取り専用解決結果の一致検証
- 7部位等の `RuntimeHardLimit`、攻撃判定予約容量、光源要求の分類・品質フォールバックを検証
- 背景4／戦闘8の光源数はデータ拒否上限にせず、`PrototypeStressTarget` の負荷試験条件として計測
- `PrototypeStressTarget` の同時50判定を、拒否上限ではなく負荷試験条件として検証
- アクションごとの最大同時判定数と、想定同時実行時の最悪ケース検証
- スキーマバージョン管理

## 4. 権威の原則

> origin: `user_explicit`（装備個体の損傷所有権とHeat/BurnCurse）、`user_approved`（権威分離）

次は常にシミュレーション側が決定する。

- 攻撃が発動したか
- 何に命中したか
- 何点のダメージか
- core `EquipmentInstance` の `Integrity`、`Deformation`、`temperature` がどう変化したか
- `Heat` と `BurnCurse` のどちらが、どの耐性を参照して適用されたか
- 疲労破壊により、どの装備個体と個体ツリーが使用不能になったか
- どの部位が破壊されたか
- 敵がHP0になったか
- ギミックが作動したか
- 何を剥ぎ取ったか

次は表示側だけで決定してよい。

- 火花の本数
- 煙の密度
- 破片の飛び方
- カメラ揺れ
- 残光
- 音の微小なピッチ差
- 背景アニメーションの間引き

表示層は、`AvatarIdentity`、装備個体の加工・スキルツリー、疲労破壊結果を書き換えない。

最小試作では疲労破壊そのものを実行しないため、上記の疲労破壊に関する権威判断は将来責務である。表示層は将来予約フィールド／イベントを参照・表示しない。

## 7. 将来の狩猟インスタンスサーバー責務

> origin: `assistant_proposal`。将来候補であり、今回の実装対象ではない

- プレイヤーと敵の正規戦闘状態
- 行動可否、クールダウン、資源

## 8. 将来の永続サービス責務

> origin: `user_explicit`（5層境界と疲労破壊対象）、`assistant_proposal`（永続サービス候補）。今回の実装対象ではない

- アカウントと認証
- `AvatarIdentity` の保存
- `MaterialJob` と既知設計・解放情報の保存
- `CombatForm` の解放・選択情報の保存
- `EquipmentInstance` の素材、形状参照、`Integrity`、`Deformation`、加工状態、疲労履歴、個体ツリーの保存
- `Appearance` の外装、主色、副色、発光色等の保存
- インベントリ
- 疲労破壊時は対象 `EquipmentInstance` とその加工・個体スキルツリーだけを破壊済みにし、`AvatarIdentity` は維持

- 報酬の確定保存

## 9. パーティクルと光源

> origin: `user_explicit`

- 視覚パーティクルは端末側だけで再生する。
- パーティクルの個別位置・速度・寿命を同期しない。
- サーバー相当層は、必要なら「効果が発生した」というイベントだけを出す。
- 光源は表示要求であり、ダメージ判定に使わない。
- 攻撃範囲は独立した軽量な当たり判定で表す。

## 10. 攻撃判定予算の責務

> origin: `user_explicit`

- プレイヤー操作由来の重要判定へ予約枠を確保し、装飾・低優先度判定が予約枠を消費できないようにする。
- ボスの必須ゲームプレイ判定と低優先度判定にも明示的な優先度を持たせる。
- 同一の所有者、作用、対象規則、時間帯、範囲として扱える判定は一つへ統合する。
- 技ごとの最大同時判定数を定義し、想定同時実行の最悪ケースを読み込み時またはビルド時に検証する。
- コンテンツ組み合わせから算出した重要判定の最悪同時数が、事前確保した予約容量を超える場合はコンテンツ検証エラーとし、本番ランタイムで初めて発覚させない。
- `PrototypeStressTarget` の50件は負荷試験で成立させる条件であり、51件以上を一律拒否するコンテンツ上限にはしない。
- ランタイム飽和は開発上の不具合としてログ・カウンター・アサートで可視化する。
- 一度受理され、資源を消費したプレイヤー技を、判定プール飽和を理由に不発にしない。既存の判定も上書きしない。
- 視覚だけの弾、火花、破片は攻撃判定プールを使用しない。

これらはローカルのプール管理・データ検証として実装候補にできるが、ネットワーク同期やサーバーを導入する理由にはしない。

## 11. 更新頻度

> origin: `user_explicit`

- 遠距離キャラクターの正規表示更新は毎秒5回以下とし、端末側で補間する。
- 温度・腐食は毎秒2～5回のシミュレーション更新とする。
- 重要な近距離戦闘を遠距離更新頻度へ落とさない。
- 描画60fpsとAI判断・継続効果の更新頻度を分離する。

`Heat` と腐食の低頻度更新は毎秒2～5回を目標とする。`BurnCurse` が瞬時適用ならイベント時に処理し、継続効果を持つ場合も `Heat` と同じ値へ統合せず独立集約する。

## 12. 試作段階で準備する境界

> origin: `user_explicit`（大規模実装とネットワーク実装の禁止）、`codex_recommendation`（将来移行を妨げない最小境界）

オンラインコードは作らず、次だけを守る。

- 安定したコンテンツIDとエンティティID
- 入力を `InputCommand` にする
- 戦闘結果を `DomainEvent` にする
- UIとアニメーションからゲーム状態を書き換えない
- 乱数生成器を注入可能にする
- シミュレーション時間を描画フレームから分離する
- 実行状態を単純な値として読み出せるようにする
- シミュレーション定義に画像・音声の実体を持ち込まない
- `AvatarIdentity`、`MaterialJob`、`CombatForm`、`EquipmentInstance`、`Appearance` を別データとして構成する
- `EquipmentLoadout` にcore装備を1個だけ構成し、その実行状態を `core_equipment_runtime_ref` から参照する
- core `EquipmentInstance` の損傷状態を表示層から直接変更しない
- `MaterialJob`／`CombatForm` はcore装備を権威元とし、Actor側は読み取り専用解決結果にする
- `Heat` と `BurnCurse` を別チャンネルとして定義・集約する
- 重要攻撃判定の予約、優先度、統合、設計時検証を行う

## 13. 試作段階で準備しないもの

> origin: `user_explicit`（ネットワーク、アカウント、永続サービスを試作へ入れない）、`user_approved`（延期する技術項目の具体化）

- 完全決定論
- ロールバック
- ラグ補償
- パケット形式
- 暗号化
- アカウントトークン
- 専用サーバービルド
- データベース
- マイクロサービス
- ネットワーク通信コード
- オンライン用の永続化実装
- 独立した武器・鎧耐久と複数装備への損傷分配
- `processing_history`、`fatigue_history`、`instance_skill_tree_state` の生成・更新・保存・表示
- `EquipmentFatigueChanged`、`EquipmentDestroyed` のイベント実装・発行・購読

これらは、オフライン試作の評価後にオンライン方式を決定してから設計する。
