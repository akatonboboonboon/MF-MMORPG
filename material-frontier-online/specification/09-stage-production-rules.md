# Material Frontier Online — ステージ制作規約

- 文書状態: レビュー草案
- 目標: 少数の実判定と再利用部品で、軽いが細部と用途を感じるステージを作る
- Gate 0状態: 閉鎖中。本文書の改訂はステージ制作開始の承認ではない

## 1. 一画面の定義

> origin: codex_recommendation

「一画面」は、基準カメラと基準ズームにおける1920×1080の表示範囲とする。

検査では次の位置で画面を採取する。

- 戦闘開始位置
- 入口と出口
- ボス初期位置
- 各ギミック位置
- カメラを横・縦に半画面ずつ移動した各地点
- ズーム変更を提供する場合は最小・最大ズーム

## 2. レイヤー分類

> origin: codex_recommendation

| レイヤー | 用途 | 当たり判定 |
|---|---|---|
| `GameplayGeometry` | 床、壁、通行不能物、戦闘障害物 | あり |
| `InteractiveGimmick` | 操作装置、危険領域、戦闘ギミック | 必要な範囲だけあり |
| `BackgroundStatic` | 背景構造物 | なし |
| `BackgroundAnimated` | 動く背景設備 | なし、物理演算なし |
| `DecorativeProp` | 配管、工具、箱、汚れ、小物 | なし |
| `ForegroundOccluder` | 手前の梁、配管、蒸気等 | なし、自動透過対象 |
| `DecalAtmosphere` | 傷、錆、濡れ、焼け、霧、光 | なし |

装飾物ごとに個別スクリプトや物理ノードを持たせない。必要な周期アニメーションは共有管理する。

## 3. 規約・制作目安と検証方法

> origin: user_explicit

各項目の強度を次のように区別する。

| 種別 | 合否への影響 |
|---|---|
| 必須規約 | 違反があればステージゲート不合格 |
| 試作上限 | 今回の試作スコープでは超過不可。正式版の製品上限ではない |
| 制作目安 | 自動的な不合格条件にしない。スクリーンショット、実プレイ、性能計測で意図と結果を審査 |

### ST-001 — 装飾物には原則として当たり判定を付けない［必須規約］

- 衝突は `GameplayGeometry` と承認済み `InteractiveGimmick` に限定する。
- 装飾タグを持つ要素に衝突形状があれば検証失敗とする。
- 見た目上の細い配管やボルトへ個別衝突を付けない。

### ST-002 — 意図しないタイル反復感を避ける［制作目安］

- 「同一タイルが3回続いたら自動失敗」とはしない。3回以上の連続は、意図しない反復感が出ていないかを確認するレビュー契機とする。
- 工場床、コンベア、配管、圧延ラインなど、用途上の規則性を示す**意図した工業反復**は許可する。
- 主要な視線領域で反復がノイズになる場合は、端部、接合部、汚れ、照明、デカール、設備の節目で変化を付ける。
- 透明または通常距離で見えない差分を、反復対策済みと見なさない。
- 合否はタイル枚数ではなく、基準距離のスクリーンショットと実プレイで「安価なコピー感」や経路の誤認がないかを目視確認して判断する。

### ST-003 — 視覚的な優先順位を作る［制作目安］

- 「画面内の大型物を必ず1個だけにする」とは解釈しない。ボス、大型炉、主タンク、重要扉、主要装置が同時に見える構図を許可する。
- 各検査画面では、その瞬間に最初に読ませたい主対象を定め、明度、動き、彩度、輪郭、配置で優先順位を作る。
- `FocalAnchor` はレビュー補助タグとして使用できるが、タグ数だけで合否を決めない。副次対象 `SecondaryAnchor` との競合をスクリーンショットと実プレイで確認する。
- 戦闘中はボス、攻撃予告、操作対象が背景ランドマークより常に高い情報優先度を持つ。

### ST-004 — 背景の動く大型物体は2個以下［試作上限］

- 画面幅または高さの25%以上を占めて動くもの、または主要シルエットを変えるものを大型と数える。
- 同じ駆動源で一体として動く子部品は1個として数えてよい。
- 座標補間、スプライトアニメーション、シェーダーで表現し、物理演算しない。

### ST-005 — 環境ギミックは一画面に最大2個［試作上限］

- 同時に有効または操作可能な独立したギミック経路を数える。
- 同じ目的と状態を共有する一組の装置は、内部的に1ギミックとして管理してよい。
- 装飾として動くだけの設備はギミックに数えず、戦闘結果も変えない。

### ST-006 — 装飾密度を可読性と過描画で管理する［制作目安］

- 装飾が移動経路、プレイヤー輪郭、敵部位、攻撃予告、ギミック状態を妨げないことを最優先とする。
- 小物の画面占有率20%は初期レビュー用の参考線にのみ使い、20%の上下だけで合否を決めない。
- 重なり、半透明、前景、パーティクルを含む過描画を計測し、性能予算を圧迫する装飾は整理する。
- 通常、最低画質、グレースケール、戦闘中スクリーンショットで情報密度を審査する。
- 小物数を増やす代わりに、用途と設備の入出力が分かる小物を優先する。

### ST-007 — 重要な足場と背景の明度差を確保する［必須規約］

- 色相だけで区別せず、明度、縁、影、輪郭を併用する。
- グレースケール表示で経路と危険境界を確認する。
- 局所コントラスト3:1を検査目安とするが、最終判断は実プレイ距離の可読性で行う。

### ST-008 — 攻撃予告と背景エフェクトの色・形を混同させない［必須規約］

- 攻撃予告専用の色、輪郭、ハッチ、脈動パターンを予約する。
- 背景の警告灯、蒸気、発光床へ同じ組み合わせを使わない。
- 通常、グレースケール、色覚補助、最低画質で検査する。
- 背景を簡略化しても攻撃予告は消さない。

### ST-009 — 前景はプレイヤーを完全に隠さない［必須規約］

- 前景とプレイヤーが重なる場合は、透過、切り抜き、輪郭表示のいずれかを行う。
- 全移動経路を走査し、プレイヤーの輪郭が完全に消える位置をなくす。
- 蒸気や煙も長時間100%遮蔽にしない。

### ST-010 — 移動経路はシルエットだけでも認識可能にする［必須規約］

- 入口、出口、主経路、袋小路の輪郭を、細かな模様や文字へ依存させない。
- テクスチャ、色、照明を除いた二値シルエットで確認する。
- 戦闘中に閉じる経路は、形状変化から判別できるようにする。

## 4. 情報表示の優先順位

> origin: user_approved

背景美術より次を前面へ表示する。

1. 即時または近日中にダメージを与える範囲
2. ボスの本体、部位、破壊状態
3. 環境ギミックの危険・操作可能表示
4. プレイヤーと将来の味方の輪郭
5. 剥ぎ取り可能位置
6. 装飾エフェクト

画質設定で1～5を削除しない。

## 5. 軽量に細部を見せる方法

> origin: codex_recommendation

### 5.1 五層構成

```text
1. 地形・移動可能領域
2. 大型構造物
3. 用途を示す中型設備
4. 小物・汚れ・配管等の装飾
5. 光、煙、音、動く背景
```

実際のゲームプレイ衝突は主に1と、一部の2・ギミックだけが持つ。

### 5.2 再利用部品

- 直線・曲がり配管
- バルブ、フランジ
- 梁、柱、手すり
- タンク、コンテナ
- ケーブル、警告灯
- コンベア、ファン、通気口

アトラス化し、左右反転、色差、汚れ、重なり順で変化を作る。同一部品を再利用しても、同じ画面構成を反復しない。

### 5.3 光と反射

> origin: codex_recommendation

- 静的な光と影は原則として背景へ焼き込む。

> origin: user_explicit

- 試作の負荷試験では背景4、戦闘8のリアルタイム光を同時に再現する。これは `PrototypeStressTarget` であり、正式版の製品上限ではない。
- 通常のステージ制作では変化が必要な光だけをリアルタイム化し、使用数を計測する。

> origin: codex_recommendation

- 小型電球は発光画像として表現する。
- 全画面リアルタイム反射を使わず、ハイライト、ぼかした反射画像、濡れた床の光筋で表現する。

### 5.4 動き

- 大きくゆっくり動く背景は2個以下。
- 小さな周期設備は共有更新する。
- 蒸気、雨、粉塵は端末側パーティクルにする。
- 画面外では停止または低頻度化する。

### 5.5 破壊表現

- 通常、損傷、破壊の状態差分を基本とする。
- 破片は端末側だけで短時間表示する。
- 常駐する物理破片やネットワーク対象を作らない。
- 戦闘に影響する破壊だけをシミュレーション状態へ持つ。

## 6. 小物の意味

> origin: codex_recommendation

小物は数ではなく用途を説明するものを選ぶ。

- 設備の入力・出力が分かる配管
- 圧力計、バルブ、検査カメラ
- 不良品箱、運搬容器
- 濡れ、腐食、焼け、摩耗の原因が推測できる痕跡
- 安全表示、製造番号、品質印

説明文を読まなくても「ここで何を扱っているか」が推測できることを目標にする。

## 7. 環境ギミック規約

> origin: codex_recommendation

各ギミックは次を満たす。

- 一つの明確な操作または発動条件
- 待機、作動、クールダウン等の少数状態
- 戦闘数値、位置、敵状態のいずれかへ実際に影響
- 発動前の明確な表示
- 攻撃予告と区別できる見た目
- 最低画質でも状態が分かる代替表示
- リトライ時に初期状態へ戻る

## 8. ステージ制作ゲート

> origin: user_approved

### Gate S1 — グレーボックス

- 移動経路、入口、出口、ボス空間、ギミック位置が成立する。
- シルエットだけで主経路を判別できる。
- OD-006が承認された場合のみ、4人が将来分散できるアリーナ余白を合格条件へ加える。未承認の間はこの項目でGate S1の合否を決めない。`origin: codex_recommendation`

### Gate S2 — ゲームプレイ

- プレイヤー、ボス、2ギミックを仮表示で配置する。
- 背景なしでも戦闘が成立する。
- 前景や装飾なしで部位を狙える。

### Gate S3 — 美術規約

- 必須規約と試作上限へ違反がない。
- タイル反復、視覚的優先順位、装飾密度は制作目安としてレビューし、懸念と判断理由を記録する。
- 装飾物に許可外の当たり判定がない。これは制作目安ではなく必須合格条件とする。

### Gate S4 — 可読性

- 通常、グレースケール、色覚補助、最低画質で検査する。
- 攻撃予告、経路、部位、ギミック状態を識別できる。

### Gate S5 — 性能

- 最大光源と対象パーティクルを有効にする。
- `08-performance-budget.md` の性能試験を通過する。

## 9. ステージ主題の扱い

飲料充填工場、製鉄・圧延区、海洋プラント等は会話中の候補である。最小試作の正式主題は未確定であり、`OD-011` の承認後に制作を開始する。
