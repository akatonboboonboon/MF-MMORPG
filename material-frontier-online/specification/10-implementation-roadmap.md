# Material Frontier Online — 実装工程

- 文書状態: レビュー草案
- 重要: **Gate 0を通過するまでゲーム本体を実装しない**
- 工程方針: 機能数ではなく、各段階の検証結果を通過条件にする
- 現在のGate 0状態: **閉鎖中**。今回の文書改訂によって自動的に開かない

## Phase 1 — 技術基盤と測定環境

> origin: codex_recommendation

### 工数制約と実装順

> origin: user_explicit

- Gate 0で試作全体の見積り単位を決め、Phase 1だけの基盤作業を**試作全体工数の15%以下**に制限する。
- Phase 1の実績または再見積りが15%へ達する前に、残る汎用基盤を後回しへ戻す。基盤完成を理由に上限を延長しない。
- 最優先の縦経路は「入力を受ける → 仮プレイヤーが移動する → 仮攻撃Aが標的へ命中する → 結果をログ表示する」とする。仮攻撃Aは正式な軽攻撃を意味せず、OD-008の決定を先取りしない。
- 将来オンライン化の抽象化、汎用プラグイン基盤、全データ型の先行実装はPhase 1へ含めない。
- Phase 1の目的は完全な基盤ではなく、プレイヤーが早期に動き、攻撃でき、後続機能を実プレイで検証できる状態を作ることである。

### 作業

- 最小のプロジェクト構造と、当面必要な命名規則だけを作る。
- 入力アクションを定義し、仮プレイヤーの移動と仮攻撃Aを最初に通す。
- 仮標的への命中結果をログまたは最小デバッグ表示で確認できるようにする。
- Phase 2までに使用する定義データだけの読み込みと検証を作る。
- 画面遷移、ログ、デバッグHUDは現在の検証に必要な最小範囲だけ作る。
- ローカル権威シミュレーションと表示を分離する。
- 部位、攻撃判定、光源、更新頻度の中央カウンターは、Phase 1で実際に存在する項目から順に追加する。
- 攻撃判定のプレイヤー予約枠と技別設計上限を検証できる最小プールを作る。50固定枠や51個目の技不発処理は作らない。
- 1920×1080のリリース相当ビルドを出力する。

### Gate 1

- 基準端末と画質設定が記録されている。
- デバッグHUDと性能ログの値が一致する。
- 現時点で定義済みの `RuntimeHardLimit` 違反を起動時に検出できる。
- 空シーンがフルHD・60fpsで安定する。
- 仮プレイヤーが移動し、仮攻撃Aを標的へ命中させられる。
- Phase 1の基盤作業が試作全体工数の15%を超えていない。
- 実際のネットワーク、アカウント、課金が含まれていない。

## Phase 2 — 共通戦闘システム

> origin: codex_recommendation

### 必須の簡易損傷モデル

> origin: user_explicit

- `Integrity` をHPとして実装し、0で機能停止させる。
- `Deformation` を0～100で必須実装し、物理被弾で増加させる。データ定義した閾値以上では、承認された単一の性能低下だけを適用する。ペナルティ対象、閾値、効果量は `OD-027` で決定する。
- プレイヤーの敗北条件はcore装備の `Integrity == 0` とし、`Deformation` 単独では敗北させない。
- 戦闘中は `Deformation` を自然回復させない。リトライまたは戦闘状態の初期化時に0へ戻し、閾値ペナルティを解除する。
- 閾値60、移動性能-15%はPhase 2の初期データ候補とするが、これは `origin: codex_recommendation` の未承認値であり、Gate 0の決定事項には含めない。Phase 2開始時または最初のプレイテストで変更できる。
- 組織スロット破壊、加工履歴巻き戻し、精密な外見変形、永久損傷は実装しない。

### core装備と将来予約項目

> origin: user_explicit

- 最小の `EquipmentLoadout` を作り、`core_equipment_instance_id` が指すcore装備1個だけを実行対象にする。
- core装備の実行状態だけがプレイヤーの `Integrity`、`Deformation`、`temperature` を所有する。Actor側へ同じ可変値を複製しない。
- `MaterialJob` と `CombatForm` のIDはcore装備を権威元とし、Actor側の参照は読み取り専用の導出値とする。
- 独立した武器耐久、鎧耐久、複数装備スロット、装備間の損傷分配は実装しない。
- `processing_history`、`fatigue_history`、`instance_skill_tree_state`、`EquipmentFatigueChanged`、`EquipmentDestroyed` は将来予約名に留め、最小試作では生成・更新・保存・UI表示・イベント発行を実装しない。

### 作業

- Phase 1の仮操作を置き換え、プレイヤー1体の移動、照準、回避、被弾、`Integrity == 0`による戦闘不能を作る。
- ダメージと当たり判定を作る。
- ステータスを素材選択時に事前計算する。
- 物理攻撃2種類を共通行動・効果から構成する。
- テスト用標的を1体用意する。

### Gate 2

- 2物理攻撃が入力、予備動作、命中、ダメージ、後隙まで成立する。
- `Integrity` と `Deformation` が別の値として変化し、定義した変形閾値による単一ペナルティが適用される。
- `Deformation` 単独では敗北せず、戦闘中は低下しない。リトライまたは状態初期化時に0へ戻り、単一ペナルティが解除される。
- core装備の実行状態が `Integrity`、`Deformation`、`temperature` の唯一の可変な正であり、Actor側に重複保持されていない。
- 技ごとの専用ダメージ処理がない。
- 攻撃判定の取得と返却が技別設計上限とプレイヤー予約の範囲で完結し、負荷状況でプレイヤー技が不発にならない。
- リトライ後に攻撃判定が残らない。
- 定常戦闘で毎フレームの不要な一時生成がない。

## Phase 3 — 素材3系統と魔法3種類

> origin: user_explicit

- パーティクルを全無効にしても戦闘結果が変わらない。
- 有効にした温度・腐食の更新が毎秒2～5回の範囲を外れない。

## Gate Playability — 遊びの成立確認

> origin: user_explicit

Gate 6の技術要件とは別に、人が実際に操作して次の6項目を確認する。自動テストや仕様上の存在だけで合格扱いにしない。

1. 移動と回避が気持ちよい。
2. 承認された2つの物理攻撃の用途が明確に異なる。
3. 3素材の差を事前説明なしでも感じられる。
4. 敵の攻撃を見て回避できる。
5. 部位を狙う意味がある。
6. 最低画質でも戦闘状況を理解できる。

確認時は、初見操作の観察、入力と被弾の記録、素材を切り替えた比較、最低画質の録画を残す。いずれかが成立しない場合はPhase 2～6の該当工程へ戻り、性能最適化だけで問題を覆い隠さない。

Gate Playabilityを通過するまで、Phase 7へ進まない。

## 9. 将来工程の順序だけを予約する

> origin: codex_recommendation

Gate 8の後に進む場合も、一度にMMO化しない。

```text
単独戦闘の評価・改修
  → ローカル複数キャラクター負荷
  → 2～4人の小規模オンライン
  → ロビーと狩猟インスタンス
  → 永続キャラクターデータ
  → ギルド・市場等のMMO機能
  → モバイル／Web等の別プラットフォーム
```

この順序は将来機能の実装承認ではない。各段階で別の仕様確認とゲートを必要とする。
