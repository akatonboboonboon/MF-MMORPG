# Material Frontier Online — パフォーマンス予算

- 文書状態: レビュー草案
- 目標: フルHDで60fps
- 注意: 正式な合否判定には基準端末の確定が必要
- Gate 0状態: 閉鎖中。本文書の改訂はゲーム本体の実装承認ではない

## 1. 予算の分類

> origin: user_explicit

性能に関する値を、次の3種類へ分離する。

| 分類 | 意味 | 扱い |
|---|---|---|
| `PrototypeStressTarget` | オフライン試作で同時に再現して負荷を測る条件 | 試作の性能試験条件。正式版の人数・容量・コンテンツ上限ではない |
| `RuntimeHardLimit` | 現在の試作ルールまたはデータ契約として、実行時に超えてはならない条件 | 読み込み前検証またはアーキテクチャ制約で保証する |
| `ProductTarget` | 正式版で保証する解像度、人数、容量、端末範囲 | Gate 8以後に別途決定する。試作値を自動継承しない |

### 1.1 PrototypeStressTarget

> origin: user_explicit

次の数値は今回指定された**試作の負荷試験条件**として保持する。これらは正式版MMORPGの製品容量でも、通常プレイ中に技を拒否するためのハード上限でもない。

| ID | 項目 | 試作負荷条件 | 検証方法 |
|---|---|---:|---|
| PST-001 | 描画目標 | 1920×1080、60fps | リリース相当ビルドを基準端末で計測 |
| PST-002 | 通常時の同時プレイヤー表示 | 8人 | 1実プレイヤー＋7表示代理 |
| PST-003 | 狩猟インスタンス | 1～8人 | 試作では通信なし。8人分の表示・更新負荷のみ再現 |
| PST-004 | 大型敵 | 1～2体 | 1体は実敵、必要なら2体目は負荷代理 |
| PST-005 | 小型敵 | 最大30体 | 固有コンテンツではなく簡易負荷代理でもよい |
| PST-006 | 破壊可能部位 | 大型敵1体あたり最大7部位 | 部位状態と描画負荷を監視 |
| PST-007 | 実判定付き攻撃 | 同時50個 | 優先度別カウンターで有効数を監視 |
| PST-008 | 視覚パーティクル | 端末側のみ | 全無効でも戦闘結果が同じことを確認 |
| PST-009 | 背景リアルタイム光源 | 4個 | カメラ内の有効要求数を監視 |
| PST-010 | 戦闘用リアルタイム光源 | 8個 | 攻撃・プレイヤー・敵由来の有効要求数を監視 |
| PST-011 | 遠距離キャラクター更新 | 毎秒5回以下 | 対象ごとの更新回数を1秒単位で記録 |
| PST-012 | 温度・腐食計算 | 毎秒2～5回 | 描画と独立したスケジューラを記録 |

試作の実コンテンツはプレイヤー1体、大型敵1体である。性能検証のためにゲーム機能を増やさず、簡易代理で上限負荷を再現する。

### 1.2 RuntimeHardLimit

> origin: user_explicit

現在の試作でハード制約とするのは、ゲームルールや決定性を守るために必要な次の項目だけとする。表示人数、敵数、光源数、攻撃判定50個はここへ含めない。

| ID | 条件 | 保証方法 |
|---|---|---|
| RHL-001 | 大型敵1体の破壊可能部位は最大7部位 | 定義データの読み込み前検証で8部位以上を拒否する |
| RHL-002 | 視覚パーティクルは戦闘結果を決定しない | パーティクルから当たり判定、AI、ダメージ、乱数へ参照経路を持たせない |
| RHL-003 | 温度・腐食のシミュレーション更新は毎秒2～5回 | 固定スケジューラで実行し、描画フレーム数から分離する |

攻撃判定プールは固定50枠のハード制約にしない。必要容量は、技別設計上限、同時使用条件、プレイヤー用予約、ボス用予約、緊急予約から事前算出する。

## 2. フレーム時間

> origin: codex_recommendation

60fpsの1フレームは16.67msである。次を初期の内部ガードレールとする。

| 指標 | 暫定ソフト予算 |
|---|---:|
| CPUメインスレッド／ゲーム処理 | P95で8ms以下 |
| GPU処理 | P95で12ms以下 |
| 総フレーム時間 | P95で16.67ms以下 |

CPUとGPUは並行して動作するため、両値を単純加算しない。この内訳はエンジンと基準端末を決めた後に見直せる。現時点のフルHD・60fpsは `PrototypeStressTarget` の合否目標であり、正式版の製品保証ではない。

## 3. 基準端末

> origin: codex_recommendation

性能ゲートを正式に通す前に、次を記録する。

- CPU
- GPU
- RAM
- OSとバージョン
- ディスプレイ解像度とリフレッシュレート
- 画質プリセット
- 電源モード
- ゲームビルドの種類

基準端末が未決定の間は、開発PCの計測値を参考値とし、正式合格にはしない。

## 4. 最大負荷シーン

> origin: user_explicit

性能試験シーンでは、次を同時に成立させる。

- 1920×1080
- プレイヤー表示8人
- 大型敵2体
- 小型敵30体
- 大型敵1体あたり7部位
- 実判定付き攻撃50個
- 背景リアルタイム光源4個
- 戦闘用リアルタイム光源8個
- 温度・腐食処理を毎秒5回
- 遠距離キャラクター更新を毎秒5回
- 環境ギミック2個を同時稼働
- 対象画質プリセットの視覚パーティクルを有効化

未実装コンテンツは次の負荷代理で置き換える。

- プレイヤー代理: スプライト、アニメーション、状態表示、低頻度位置更新のみ
- 大型敵代理: 同程度の描画面積、7部位、簡易移動のみ
- 小型敵代理: 同程度の描画、簡易更新、攻撃は行わない
- 攻撃代理: 実際の軽量当たり判定と寿命だけを持つ

負荷代理を正式コンテンツとして数えない。

## 5. 測定手順

> origin: codex_recommendation

1. デバッグ実行ではなくリリース相当ビルドを使用する。
2. 最大負荷シーンを60秒間ウォームアップする。
3. 10分間の戦闘シーケンスを3回計測する。
4. 平均、P50、P95、P99、最大フレーム時間を保存する。
5. CPU、GPU、物理、スクリプト、描画、メモリ、生成オブジェクト数を個別記録する。
6. 33.3msを超えるフレームの件数と原因を記録する。
7. ウォームアップ後にメモリ使用量が単調増加していないことを確認する。
8. 最低画質でも別途実行し、攻撃予告や操作対象が消えていないことを確認する。

## 6. 合格条件

> origin: codex_recommendation

- P95総フレーム時間が16.67ms以下。
- 1秒以上継続する60fps未満の区間がない。
- 3回の反復でメモリリークが認められない。
- `RuntimeHardLimit` 違反がない。
- `PrototypeStressTarget` の同時負荷条件を欠落なく成立させている。
- パーティクル無効時にも戦闘結果と乱数結果が同じ。
- 低画質化しても攻撃予告、当たり判定、部位状態、ギミック判定が維持される。
- ロードやシーン切替の一時停止は戦闘中計測と分けて記録される。

いずれかを満たさない場合は性能ゲート不合格とし、次のコンテンツ工程へ進まない。

## 7. 実判定付き攻撃とリソース予約

> origin: user_explicit

- 中央カウンターは `PrototypeStressTarget` の計測と `RuntimeHardLimit` の監視を別項目として表示する。
- 実判定リソースは、少なくとも `PlayerCritical`、`BossCritical`、`Environment`、`LowPriority` の優先度へ分ける。
- プレイヤー攻撃用とボスの必須攻撃・予告用に予約枠を確保し、装飾攻撃や低優先度環境判定が予約枠を消費できないようにする。
- 各 `ActionDefinition` に技別の最大同時判定数を定義し、1回の技が無制限に判定を生成できないようにする。
- 意味、対象集合、時間区間が同じ範囲攻撃は一つの判定へ統合する。統合で結果が変わるものはまとめない。
- ステージと行動セットの組み合わせから最悪同時数を事前計算し、予約容量を超える定義は開発時エラーとしてビルドまたはデータ検証を失敗させる。
- 本番相当ビルドでは、負荷状況を理由にプレイヤー技やボスの必須攻撃を不発にしない。資源消費やクールダウン結果もプール空き状況へ依存させない。
- 予期しない逼迫時は、低優先度判定の新規生成停止、意味が同じ範囲の統合、装飾・低優先度処理の削減を先に行う。既に成立した重要攻撃を上書きしない。
- 設計漏れへ備え、通常予約とは別の事前確保済み緊急枠を持つ。緊急枠使用はエラーテレメトリとして記録し、正式合格条件では0回とする。
- `PST-007` の50個は試験時に成立させる同時負荷であり、「51個目を拒否する分岐」を実装する根拠にはしない。
- パーティクル、火花、煙、破片は当たり判定、AI、ダメージ計算へ使わない。
- 温度・腐食は経過時間を積算して更新し、2Hzと5Hzで最終結果が大きく変わらない式にする。
- 遠距離対象は毎秒5回以下で正規位置を更新し、表示は補間する。

## 8. 毎フレーム処理を避ける対象

> origin: codex_recommendation

- スキルツリー全走査
- 合金組成からの再計算
- 温度・腐食・疲労の継続計算
- 遠距離AI判断
- 背景装飾の個別スクリプト
- 報酬表の事前展開
- パーティクルの戦闘照会

定常戦闘中、ゲーム側スクリプトによる毎フレームの一時オブジェクト生成を原則なくす。

## 9. 予算超過時の最適化順

> origin: codex_recommendation

まずプロファイル結果に基づき、次を行う。

1. オブジェクトプール
2. ステータスの事前計算
3. 同種継続効果の集約
4. 空間検索の範囲限定
5. バッチ化とアトラス化
6. 画面外・遠距離更新の間引き
7. 不要なイベント購読と残留オブジェクトの除去

画質を落とす場合は次の順とする。

1. 戦闘に無関係な火花、煙、粉塵、蒸気の密度と寿命
2. デカール、残光、細かな環境粒子
3. 背景設備のアニメーション更新頻度
4. 背景リアルタイム光を焼き込み発光へ置換
5. ぼかし、反射、色収差などの装飾効果
6. 遠距離プレイヤーの装飾レイヤーとアニメーション頻度
7. 戦闘用の装飾光を軽量な発光スプライトへ置換
8. 小型敵の装飾アニメーションと付随SE数

## 10. 劣化させない項目

> origin: user_explicit

- 攻撃予告の形、輪郭、危険範囲
- プレイヤー入力
- 移動、当たり判定、ダメージ判定
- ボスHP、部位状態、HP0判定
- 環境ギミックの判定
- 剥ぎ取り結果
- フルHDという合否時の解像度
- 指定された `PrototypeStressTarget` の試験条件

これらを維持できない場合、画質設定で隠さず性能不合格として扱う。
